FrameCraft — Technology Overview

One sentence in, one brand-aware motion-graphics video out.

FrameCraft plans a film as a structured narrative, writes the HTML/CSS/GSAP for every scene using an LLM, self-validates against ~14 runtime-safety and aesthetic rules, and hands the result to a deterministic renderer for MP4 output. Zero designers in the loop.

doc v1 · author-side pipeline through scene render · code revisions v4 → v10

§1Executive summary

A brand video traditionally costs weeks and tens of thousands of dollars: a designer, an animator, a copywriter, and rounds of feedback. FrameCraft compresses that into two LLM-driven passes and ~$0.10 per 30-second film, with every scene authored as bespoke HTML/CSS/GSAP motion graphics.

The technology has three distinctive characteristics:

  1. Story-first planning. A dedicated LLM pass produces a Story Bible for the whole film — thesis, weighted pillars, scene-level copy ledger, data exhibits, recurring motif — before any HTML is written.
  2. LLM-as-motion-designer. Each scene is authored in three LLM passes (design brief → HTML execution → elevation polish) against a rich motion vocabulary and a brand-aware palette.
  3. Self-validating output. Every scene is rejected if it contains any of ~14 known runtime or aesthetic failure modes. The LLM gets one retry; on second failure, a rich-safe fallback renders the scene deterministically from the bible's copy ledger — the film never ships a broken scene.

The output is a Hyperframes-compatible project: 7 scenes × ~400 lines of HTML, 20+ animated elements per scene, 25+ GSAP tweens per scene, a validated MP4.

§2The problem FrameCraft solves

Today's workflow

  • Brief → creative director → storyboard → rounds of revisions
  • Designer builds in After Effects / Figma → manual export
  • Each copy change = full re-authoring pass
  • Brand variants = linear cost multiplier
  • Typical turnaround: 1–3 weeks per 30s film

With FrameCraft

  • Brief → CLI → MP4 in ~8 minutes
  • Every scene authored as code — diffable, versionable, testable
  • Copy change = edit plan.json → framecraft from-plan … --render
  • Brand variants = --n-variants 5
  • Cost per 30s film: ~$0.05–0.15

This doesn't replace top-tier bespoke work, but it collapses concept→draft and serves every "we need a 30-second film for X by Thursday" situation.

§3Architecture

FrameCraft runs three sequential phases — Planning, Authoring, Assembly & Render — across eight stages. Red badges are LLM-driven; charcoal badges are deterministic. The green fallback path activates when the validator rejects two LLM attempts.

Why a dedicated story pass?

Early prototypes authored each scene in isolation — the LLM invented copy from thin block_props, couldn't reference neighbor scenes, and produced films that felt like "a list of facts animated". Introducing the bible as a first-class planning step produced the biggest quality jump in the project (v5 vs v4 below) because every downstream stage now has global narrative context.

①Story Bible

Module
src/framecraft/story_bible.py
Model
Gemini 2.5 Pro (Vertex AI) · JSON-mode · no response_schema
Output
.framecraft/story-bible.json
Latency
~15s · ~$0.02 per run · cached per situation

The bible is a full film treatment in structured JSON. One LLM call, then pydantic validation, then a one-shot retry if the output fails cross-reference checks (orphan pillars, duration mismatches, char-budget overruns).

What the bible captures:

  • Thesis — one sentence: the single takeaway for the viewer.
  • Pillars (2–7) — key messages, each tagged with tier (hero support context), a weight 1–5, and an anchor flag for the scene that carries the film's signature moment.
  • Scenes (2–8) — each with a narrative role (hook, hero-beat, evidence, proof-point, resolution, brand-lockup, transition), a duration_s, a tone, a full copy ledger (every text element with max_chars budget), and entry_motif + exit_motif handoffs.
  • Exhibits — data objects used by scenes: comparison_table, line_curve, bar_chart, stat_grid, checklist, timeline.
  • Motif arc — a recurring visual element (shield, logo) whose state progresses across scenes: "0% drawn" → "three edges visible" → "complete emblem".

Cross-reference invariants enforced by pydantic:

  • Every pillar id referenced in scene.carries must exist.
  • Every exhibit id in scene.exhibit_ids must exist.
  • Every hero-tier pillar must appear in at least one scene.
  • Scene indices must be contiguous from 0.
  • Sum of scene.duration_s must be within ±0.5s of total duration.
  • At least one scene must have is_anchor: true.
  • Hero-tier scenes must carry at least one pillar.
  • Every copy_item.text length ≤ max_chars × 1.15 (15% LLM grace).

③Per-scene Author

The author runs three chained LLM passes with a fresh context each time. The prompt weighs in at ~30,000 characters — a significant investment that delivers proportional output quality.

Figure 2 · The per-scene author loop. Pass 2 is the only gated pass — failure triggers a single retry with the specific validator error fed into the next prompt. Pass 3 is best-effort (keeps the Pass-2 HTML if elevation fails). Two-time failure yields to the rich-safe fallback that builds the scene deterministically from the bible's copy ledger.

Pass 1 · Design brief

The LLM writes a short text-format brief committing to:

  • NARRATIVE — what story this scene tells inside the film.
  • LAYOUT — zone assignments (Z-TL, Z-BR, Z-C, etc.) for primary content.
  • SIGNATURE_MOMENT — the one hero beat this scene carries.
  • COMPONENTS_USED — 2+ from the component vocabulary (stat grid, comparison table, checklist, timeline, number counter, etc.).
  • ELEMENTS — 20+ named elements with role, zone, visual treatment.
  • COPY — exact copy ledger, drawn verbatim from the bible.
  • MOTION_MOMENTS — 6+ beats with timing and technique.
  • TIMELINE_STRUCTURE — the nested sub-timelines to build.
  • COLOR_DEPTH — exact HEX values (from the brand system when present).
  • DOMAIN_VOCABULARY — 2–3 industry-specific motifs.
  • PACING — the rhythm (staccato / legato / crescendo-to-beat).

Pass 2 · HTML execution

Given the brief, the LLM writes the complete <template> block — scoped CSS, inline SVG, GSAP timeline with nested sub-timelines, all under strict density floors:

  • ≥ 20 animated DOM elements
  • ≥ 25 GSAP tween calls
  • ≥ 6 distinct motion techniques across ≥ 3 categories
  • ≥ 3 atmospheric layers (grain, orbs, vignette, scanlines, …)
  • ≥ 2 nested sub-timelines via gsap.timeline() + tl.add(subTl, t)
  • Exactly 1 signature moment
  • 3-phase timeline (entry → evolution → exit)
  • Every primary element gets at least one tween; every atmospheric element a continuous or breathing one

Pass 3 · Elevation polish

The LLM reviews its own Pass-2 output and adds without removing:

  • ≥ 2 new DOM elements
  • ≥ 4 new tweens
  • ≥ 1 new technique not yet used
  • A secondary ambient motion or signature detail that was missing

If the elevated output fails validation, we keep the Pass-2 result (elevation is best-effort).

④Validator gate

Module
src/framecraft/rendering/llm_author.py::_validate()
Nature
Regex + structural checks · purely deterministic · no LLM

The validator is the single most important quality lever in the system. It represents every failure mode we've seen an LLM produce, encoded as a hard rejection. On failure, the author retries once with the specific error fed back in the prompt. On second failure, we fall back.

See §Validator rules catalogue below for the full list of 14 rules.

⑤Rich-safe fallback

Module
src/framecraft/rendering/rich_fallback.py
Nature
Deterministic Python · produces ~14 animated elements + 14 tweens · always passes validator

When a scene can't get past the validator in two attempts, instead of shipping a broken scene or collapsing to a single-opacity fade, we render a bible-aware rich template using the scene's copy ledger (eyebrow, headline, subhead, bullets, tagline, CTA) and the brand's palette.

The fallback composition:

  • Cream/dark bg with two blurred brand-colored orbs
  • SVG feTurbulence grain layer
  • Radial vignette breathing
  • Conditional eyebrow-tick next to eyebrow label (only when eyebrow copy exists)
  • Staggered char/word entries with power3.out
  • Ambient camera push over full scene with sine.inOut
  • Content fade + y-shift on exit (no full-bleed wipe panel)

This was the biggest UX-coherence fix in the project: in early versions, scenes that failed LLM authoring collapsed to a boring 2-element opacity fade, and one broken scene ruined the film. The rich-safe fallback keeps scene density proportional even under LLM failure.

⑥Assembler

Module
src/framecraft/assembler.py

Writes the final files to disk:

  • compositions/scene-NN-<block_id>.html — one per scene
  • index.html — root composition wiring everything together
  • plan.json — the plan with polished.html cache written back so from-plan can skip LLM calls on re-renders
  • hyperframes.json, meta.json — Hyperframes project metadata
  • .framecraft/story-bible.json — the bible for the run
  • .framecraft/assembler-traces/scene-NN.json — per-scene token / elapsed / cache metrics

For each scene, the assembler routes through a decision tree:

polished["html"] already cached?
   └─ yes → reuse (1 hit)
   └─ no  → full_polish enabled + real provider?
              └─ yes → call LLM author (possibly falls back)
              └─ no  → Python native template (legacy)

⑦Lint + repair

Module
src/framecraft/lint.py, lint_policy.py

Runs npx hyperframes lint against the assembled project. Findings are classified:

  • FATAL — compiler-rejection findings (never expected to pass in practice).
  • LLM_REPAIRABLE — e.g. overlapping clips in a single scene — we re-author that scene with the error in context.
  • INFORMATIONAL — e.g. caption-exit kill-tween suggestions; logged but not blocking.

Findings trigger a repair loop: at most one re-authoring pass per scene. If lint still fails after repair, the run aborts with a clean exit code and a human-actionable message.

⑧Render

Module
src/framecraft/cli_render.py → npx hyperframes render
Engine
Hyperframes uses headless Chromium to capture exact-time frames, then ffmpeg encodes MP4
Latency
~3 min for a 30s @ 30fps 1080p render (6 parallel workers)

Render is deterministic: Hyperframes advances virtual time, captures each frame, muxes with ffmpeg. No wall-clock timers, no requestAnimationFrame (that's why the validator forbids them in scene JS).

Figure 4 · The rendering pipeline. Hyperframes compiles the project, splits the 960-frame capture range across 6 parallel headless-Chromium workers, seeks each worker to exact virtual times, screenshots the DOM at each tick, then hands the frame sequence to ffmpeg for H.264 mux. The validator's determinism contract (right) is what makes step B's exact-time seeking reliable.

§4Validator rules catalogue

Every rule corresponds to a failure mode observed in real LLM output during development. Each rule name is the reason string the author sees on rejection.

RuleCategoryWhat it catches
Date.nowRuntimeNon-deterministic time source; render capture can't reproduce frames.
Math.randomRuntimeNon-deterministic; different frames pick different values.
fetchRuntimeNetwork during render is forbidden; hangs the capture engine.
setTimeout / setIntervalRuntimeWall-clock timers don't respect virtual-time advancement.
requestAnimationFrameRuntimeTied to real-time display refresh; breaks deterministic seek.
performance.nowRuntimeAnother wall-clock source.
emoji_font_apple / emoji_font_segoeRuntimeEmoji fallback font-family stacks; Hyperframes compiler can't map them deterministically.
gsap_infinite_repeatRuntimerepeat: -1 loops — break exact-time frame seek; forces finite repeat: Math.floor(duration / cycle) - 1.
template_literal_selectorRuntimePatterns like gsap.to(`${scope} .x`, …) — the HTML bundler's CSS parser crashes on these; forces hardcoded strings.
clip_class_styledAestheticCSS rules targeting .clip directly (Hyperframes' visibility-control class). A scoped [data-composition-id="…"] .clip { position: absolute; } stamps layout onto every timed element — scenes render blank with everything at (0,0).
overlapping_clips_same_trackStructuralTwo .clip elements with the same data-track-index and intersecting time ranges — Hyperframes can't render the overlap correctly. Rule forces unique tracks per element.
long_text_in_svgAestheticSVG <text> node carrying >12 chars — gets clipped by the parent SVG's viewBox. Famous bug: "GUARANTEED LIFETIME COVE…"
long_text_over_shapeAestheticHTML <div class="ribbon-text|shield-label|chip-text|hexagon-text|…"> carrying >10 chars. The text extends past the shape's silhouette and becomes invisible against the canvas bg.
full_bleed_exit_panelAestheticElements named exit-panel, cover-panel, wipe-panel, scene-cover, full-wipe — a solid panel wiping in on exit creates a visible flash between scenes.
Structural requiredsRuntimeEvery scene must contain: GSAP CDN script, <template id="…"> wrapper, data-composition-id matching the comp id, window.__timelines[...] = tl registration, paused timeline.

§5Story Bible schema

Defined in src/framecraft/schema.py. All models are pydantic v2; the bible ships as a top-level StoryBible with nested Pillar, SceneBrief, CopyEntry, Exhibit, MotifArc children.

StoryBible {
  version:       1
  situation:     str                      # the original user input
  thesis:        str (10–280)             # single sentence
  audience:      str
  overall_tone:  str
  brand_keyword: str | null               # "aia", etc. — triggers brand injection
  duration:      float > 0
  pillars:       list[Pillar]   (2–7)
  exhibits:      list[Exhibit]  (≤12)
  motif:         MotifArc | null
  scenes:        list[SceneBrief] (2–8)
}

Pillar {
  id:     str   ([A-Za-z0-9_\-]+)
  claim:  str   (3–140)
  tier:   "hero" | "support" | "context"
  weight: int   (1–5)
  anchor: bool                           # "if you watched only 5s, this is the moment"
}

SceneBrief {
  index:         int
  role:          hook | hero-beat | evidence | proof-point |
                 resolution | brand-lockup | transition
  duration_s:    float
  tier:          hero | support | context
  is_anchor:     bool
  carries:       list[str]               # pillar ids
  tone:          warm-restrained | confident-punch | data-precise |
                 tender-close | energetic-pivot
  entry_motif:   str                     # "cream fade", "light sweep from right"
  exit_motif:    str                     # handoff string for next scene
  copy_items:    list[CopyEntry]
  exhibit_ids:   list[str]
  narrative:     str
}

CopyEntry {
  id:        str
  text:      str
  max_chars: int | null                  # enforced with 15% grace
  kind:      eyebrow | headline | subhead | body | bullet |
             stat_value | stat_label | caption | cta | tagline
}

Exhibit {
  id:    str
  kind:  comparison_table | line_curve | bar_chart |
         stat_grid | checklist | timeline
  # kind-specific fields follow — e.g.:
  # comparison_table → columns, rows, highlight_column?
  # line_curve       → x_label, y_label, series, style, area_fill
  # bar_chart        → bars, unit, orientation
  # stat_grid        → cells: [{value, label, chip?}, …]
  # checklist        → items
  # timeline         → milestones: [[time_label, event], …]
}

MotifArc {
  id:                 str
  description:        str
  scene_progression:  dict[str, str]     # "0" → "0% drawn", "2" → "half formed"
}

§6Data exhibits

Exhibits are first-class typed data objects. The bible declares them; the LLM author renders them via a structured rendering vocabulary embedded in the author prompt. A scene references one or more by id.

stat_grid

3–6 cells of {value, label, chip?}. Rendered as a CSS grid of rounded cards, each with staggered entry and its own ambient breathing motion.

comparison_table

N columns × M rows with a highlight_column. Rendered as CSS grid; the highlighted column gets a tinted background and a winner chip.

line_curve

SVG path with stroke-dasharray draw-in animation (Catmull-Rom → cubic Bézier), optional area-fill gradient, milestone dots, axis labels. The "sophisticated curve" rendering primitive.

bar_chart

Vertical or horizontal bars with scaleY/scaleX stagger entry. Value labels count up via gsap.onUpdate.

checklist

4–6 rows of (SVG checkmark + text). Ticks stroke-draw, text slides in from right.

timeline

Horizontal line + milestone dots. Line scales left-to-right, dots pop with back.out(1.8), labels fade alternating above/below.

Each exhibit type has 4–8 rendering instructions in the author prompt covering CSS structure, GSAP pattern, and the "don't pack everything into 0.3s" timing discipline.

§7Brand system injection

When the user's style seed or situation contains a known brand keyword, the matching brand system block is appended to both the Story Bible system prompt and the author system prompt. The block ships verbatim — no paraphrasing — so the LLM sees exact HEX codes, typographic direction, motifs, tagline, and verified product facts.

Current brand: aia — palette (#D31145 primary, #F5DDE0 card, #FAF4EE bg, #1A1A1A ink, #E86C29 callout), tagline ("Healthier, Longer, Better Lives."), motifs (first-aid cross, shield, heart+cross, ribbon badge), and 9 verified product facts (S$2M limit, 13-month pre/post, 380+ AQHP specialists, deductible-waiver pass, etc.).

Adding a new brand is a single entry in _BRAND_SYSTEMS — no pipeline changes needed.

§8Scene-to-scene seams

The bible declares an entry_motif and exit_motif per scene (short text descriptions). The author prompt for scene N receives:

  • prev_scene.exit_motif — the motion the previous scene ends on
  • this.entry_motif — what this scene should start with (should match or intentionally contrast prev)
  • this.exit_motif — the motion this scene should end on
  • next_scene.entry_motif — the handoff the next scene expects

This is how the film reads as directed rather than a slideshow. A light-sweep-right exit in scene 2 → light-sweep-right entry in scene 3 produces a visual baton-pass. The motif arc gives the viewer a through-line: a shield emblem that progresses from outline to complete across scenes 2 → 5.

§9Motion vocabulary

A ~40-item menu embedded in the author prompt, organized in 5 categories:

  • Typography — char-mask reveal, word stagger with blur, variable-weight swell, kerning collapse, split-flap flip, outline-to-fill, gradient-text-fill sweep, tracking pulse, stacked typography, kinetic number, word-replacement cascade.
  • Reveals & masking — clip-path horizontal wipe, vertical blind, radial iris, light sweep, spotlight tracker, ink-stamp SVG reveal, diagonal band wipe.
  • Atmospheric layers (scene needs ≥3) — feTurbulence grain, radial gradient orbs, backdrop-filter blur pull, depth-of-field rack, vignette breathe, scanlines, particle dot grid, mist/haze, light rays, chromatic aberration, film-burn flicker, noise mesh overlay.
  • Structural & 3D — camera push (always), parallax split, 3D perspective tilt, camera dolly, Z-axis depth stack, orbit motion, accent bar draw, SVG stroke-draw, geometric wipe transition.
  • Signature / hero moments — shield assembly, logo lockup, stat-card cascade, ripple pulse, typewriter cascade, bar-chart rise, timeline draw, spotlight reveal, icon morph-swap, mass particle explosion, hero word swap, ink wash.

The prompt demands ≥ 6 techniques across ≥ 3 categories per scene, with exactly 1 signature move.

§11Cost & latency

Per-run cost (30-second film, 7 scenes, Gemini 2.5 Pro)

Story Bible pass~$0.015
Per-scene author (3-pass × 7)~$0.07
Elevation passes (× 7)~$0.02
Retry overhead (~20% scenes)~$0.015
Total~$0.10–0.15

Per-run latency

Bible (1 LLM + retry overhead)~20s
7 × 3-pass author~6 min
Hyperframes render (960 frames, 6 workers)~3 min
Lint + repair~5s
Total~9–12 min

Caching: the bible is persisted to .framecraft/story-bible.json and each authored scene's HTML is written back to plan.json as scene.polished.html. framecraft from-plan <plan.json> re-renders with zero LLM calls — useful for tweaking copy or re-rendering at different quality.

With --n-variants 5, the bible is reused across variants and only the per-scene authoring repeats — so 5 variants cost ~4× a single run, not 5×.

§12Tech stack

Python side

  • Python 3.11 / 3.12
  • pydantic v2 — schema + validation (bible, scene graph, copy entries)
  • typer + rich — CLI
  • google-genai — Vertex AI Gemini (with service-account auth)
  • anthropic — Claude SDK (alternate provider)
  • jinja2 — native-block templates
  • beautifulsoup4 — HTML parsing for lint repair
  • pyyaml — framecraft.yaml project config
  • pytest + pytest-recording — tests, including recorded-LLM fixtures

Node side (via npx)

  • Hyperframes — composition / capture / render engine
  • GSAP 3.14 — animation (loaded from CDN inside each scene)
  • Chromium (headless, via Hyperframes) — frame capture
  • ffmpeg — MP4 mux

Determinism contract: everything the LLM writes must be reproducible given the same input. No real-time clock, no RNG, no network, no non-deterministic fonts. Violations are rejected by the validator.
